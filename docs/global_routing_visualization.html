Global Routing 현재 구조와 DFF 실패 지점

이 페이지는 현재 global PnR이 child layout candidate를 배치하고 module 간 route를 잇는 흐름을 한 화면에서 보기 위한 설명용 시각화다. 핵심 문제는 sequential output을 단일 route point로 모델링하면서 latch 내부 feedback net과 외부 global route의 경계가 흐려지는 데 있다.

3. Route Search State

현재 문제 요약

  • PhysicalPort가 논리 output 위치와 routing access 위치를 충분히 분리하지 못한다.
  • sequential output을 내부 redstone tap으로 잡으면 RS latch feedback net에 외부 route가 합류한다.
  • q torch 자체를 source로 쓰면 global router가 route를 못 찾는 경우가 많다.
  • repeater stub를 미리 하나 고르면 target 방향과 안 맞아서 route가 막힐 수 있다.
  • shelf placement가 단순해서 DFF처럼 latch를 이어 붙이는 구조에서 포트 alignment가 우연에 의존한다.

권장 다음 모델

PhysicalPort를 단일 위치가 아니라 복수 access 후보로 확장한다. 예를 들면 logical_position, access_points[], requires_isolation, cost를 둔다.
global router는 source access와 sink access 조합을 같이 고른다. 그러면 q torch, redstone tap, repeater-isolated tap 중 target에 맞는 후보를 선택할 수 있다.
global placement는 route 실패 시 spacing, y alignment, candidate 선택을 재시도할 수 있어야 한다. 지금처럼 shelf placement 한 번으로 끝내면 route 실패를 회복하기 어렵다.
